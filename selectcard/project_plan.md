# 杀戮尖塔：选牌推荐 AI (胜率预测模型) 实施计划

## 核心目标
利用 700 万局 Google Drive 上的历史对局压缩包（JSON 格式），训练一个基于深度学习（类似 CTR 预估/离线强化学习）的模型。
模型的输入为：`[当层卡组状态,当层遗物状态, 当前层数, 候选卡牌]`
模型的输出为：`预测当前 Act 的存活率 (即能否顺利击杀当前区域的关底 Boss，0.0 ~ 1.0)`
游戏内应用策略：分别推演选择各候选项目后的假想后续状态，推荐当前 Act 存活概率最高的一项，坚决杜绝前期盲目“听牌”。

---

## 实施路线图

### 第一阶段：数据流水线与状态重放 (Data Pipeline & Replay)
*由于JSON数据记录的是结局（终盘形态），我们需要写一个回放器（Simulator），在每一层进行时光回溯，重构出当时的动态卡组。*

1. **确立起手盘**: 根据 `character_chosen` 初始化基础卡牌（如打击、防御、专属技能牌）。
2. **构建时光机**: 提取对卡组和遗物有影响的事件日志（打上时间戳 `floor`）：
   - `card_choices`: 拿牌（加牌）
   - `items_purged` / `items_purged_floors`: 商店与事件删牌（减牌）
   - `campfire_choices`: 篝火升级（移除原牌，加入 `+1` 牌）
   - `event_choices`: 问号事件的复杂增删改变
   - `relics_obtained`: 遗物获得节点
3. **特征生成切片**: 当时间线推演到某一个 `card_choices` 节点时，暂停推演，截取当前这一刻的卡组、遗物、血量作为**输入特征**。**Label 设定**：不再使用非黑即白的整局最终胜负，而是将其后“是否成功击杀当前 Act 的关底 Boss”作为唯一目标，引导模型学习高手前期的生存保命意识，同时避免引入复杂的层数衰减计算。
4. **性能优化**: 针对包含几百万局的 `.gz` 文件，开发流式处理脚本（Streaming Generator），解析完毕即写入 Parquet 格式，防止内存溢出。

### 第二阶段：特征工程与模型架构 (Set Transformer)
放弃无法捕捉复杂体系联动的传统推荐架构，全面切换至基于 Attention 机制的 **Set Transformer**，将卡组和遗物视为“Token 序列 (句子)”，极大地提升卡牌间的 Synergy (套路、化学反应) 的理解能力。

1. **序列特征化 (Deck-as-a-Sequence)**：
   - 输入序列构造：`[CLS] + [Card_1] + [Card_2] ... + [SEP] + [Relic_1] ...`
   - **同名卡牌聚合去重 (Count Encoding)**：为解决 Transformer 的 $O(N^2)$ 计算时间瓶颈，卡组中的同名同级卡不再作为独立 Token 输入。引入数量编码：`Token_Embedding = [Card_ID] + [Upgrade_Level] + [Count_ID]`。这使得多张相同的牌只占用 1 个序列位置，极大缩短总 Token 长度，提升实机推演速度。
   - **关于卡牌升级的解耦编码**：不再为升级牌单独创建 ID，而是修饰词相加：`Token_Embedding = [Card_ID_Embedding] + [Upgrade_Level_Embedding]`。对于特例“灼热打击+N”，只需传入对应的 `Level_N` Embedding 即可，优雅化解词表膨胀问题，泛化性极强。
   - **无位置编码 (Permutation Invariance)**：由于卡组是洗牌后的无序集合，主动去掉位置编码 (Positional Encoding)，使网络成为纯粹的 Set Transformer，完美适配《杀戮尖塔》机制。
2. **连续与全局特征 (Global Context)**：
   - 层数 `floor`、当前血量 (HP)、当前金币、当前进阶等级 (A15+) 以及**目标关底Boss / 下一个精英类型**。
   - 这些全局标量输入小型 MLP 映射后，作为 `[GLOBAL_STATE]` Token 并入序列，或在 `[CLS]` 输出后拼接参与最终预测。
3. **数据清洗与异常对局处理**：
   - **进阶等级过滤**：只选取**进阶等级 15 及以上 (A15+)** 的对局数据进行训练。过滤 A15+ 能够倒逼模型学习大局端最极致的选牌与运营决策。
   - **防跑路/异常退出机制 (Abandon/Quit/Crash)**：如果一局游戏非正常死亡或通关（表现为 `victory: false` 但缺失 `killed_by` 字段，或者由于玩家放弃导致 `floor_reached` 低于实际卡组数据），该局最后所在那层（及由于控制台/放弃导致的无效结果）将被标记为脏数据直接丢弃。我们最多只截取他正常游玩并符合逻辑的*前一章*的选择，或者对于难以判定有效生存 Label 的残局直接全盘丢弃，避免模型学到“因为玩家在第 15 层拔线，导致第 10 层的胜率被判定为 0”这种冤案。
   - **作弊与控制台数据排查**：控制台 (Console) 本身的操作不会在 JSON 中显式记录（比如敲了 `add gold 1000`）。但为了防范数据污染，**第一道防线**是直接丢弃带 Mod （或者带异常字符如自定角色）的对局；**第二道防线**是在时光回放器中加入“状态平账校验”：由于每层的事件 (如金币获取、掉血量) 都在 JSON 中，如果在回溯时，发现两层之间的金币/卡组突然凭空多出/减少而无相关事件记录（即物理平账失败），则直接判定此局遭到了修改，整局数据标为污染并抛弃。
   - **绝对过滤要求**：直接丢弃所有包含**七彩棱镜 (Prismatic Shard)** 的对局数据。

### 第二阶段（核心思路转变）：存活期望评估网络 (Survival-Value Network)
由于商店、事件等场景具有复杂的选择维度，直接评估单卡显得局限。因此，我们将模型设计为**当前状态生存期望评估器 $V(State)$**：
- **模型输入**：`[卡组序列 Token, 遗物序列 Token, 药水状态, 当前层数, 当前金币, 当前血量, 当前进阶等级, 目标近期高危节点 (精英/Boss), 最近商店/营火距离]`
- **模型输出**：`该状态下顺利击杀当前 Act 关底 Boss 的概率期望 (0.0 ~ 1.0)`
- **决策原理 (Transition & Evaluation)**：遇到任何选择时，我们在内存中**推演**该选择执行后的“假想新状态 (Next State)”，然后将所有候选假想状态输入模型，取评分最高的作为决策。
  - **全物品通用**：这种推演打分的机制不仅适用于“三选一”的卡牌奖励，同样完全无缝适用于**遗物选择（如 Boss 遗物、事件遗物）**和**药水选择**，只要把对应的物品 Token 塞入假想状态即可。
  - **商店多选策略 (Greedy Search)**：面对商店里琳琅满目的卡牌、遗物和药水，无需且不能穷举所有购买组合（会导致推演引发组合爆炸）。只需采用**贪心搜索 (Greedy Search)**：每次比较“只买当前单件物品”后的 $V(S')$，买下评分提升最大的那件；扣除金币更新状态后，再重复此过程。直到所有可买物品的 $V(S')$ 都不如“什么都不买（保留当前状态与金币）”的分数高时，即停止消费。这种做法完美统一了所有的评判逻辑。

针对随机性事件（如拿到**战纹涂料**、**星盘**、**变化**、**黑桶**）：由于抉择前 AI 无法预知具体结果，传统的蒙特卡洛多次推演会引发计算爆炸。因此采用**构建虚拟 Mask Token** 的方案直接让网络学习“盲盒期望”：
  - **定义虚拟 Token**：在特征字典中引入特殊的 `[RANDOM_UPGRADE]`（随机升级）、`[TRANSFORM_CARD]`（变化）等 Token，代表“拥有一次未知的强化/变化权利”。
  - **训练阶段 (数据遮蔽)**：在解析 JSON 战报重构卡组时，如果发现由盲盒事件造成的卡牌升级/变化，我们在训练时有一定概率（如 50%）**隐藏**真实结果状态，而是将卡组回滚到盲盒开启前的状态，并在序列中硬塞入对应的虚拟 Token（例如推入 2 个 `[RANDOM_UPGRADE]` 代表战纹涂料），随后打上和真实结果相同的胜率 Label 去训练。这样强制模型直接拟合评估“盲盒事件”的数学期望。
  - **实机推理阶段 (One-pass Inference)**：遇到该选项时，不再需要在内存中随机抽卡模拟 5 次求平均，而是直接将当前的假想状态拼入对应的虚拟 Token 丢给模型。网络只需进行**仅一次**前向传播，其输出在数学上会自然收敛于所有可能情况的期望胜率，将其与“不买”的分数对比作决策即可，彻底解决实机推演的算力瓶颈。

### 第三阶段：AIPlayspire 接口集成 (Choice-Driven API)
1. **模型导出与服务化**: 
   - 将 PyTorch 训练稳定的模型导出为轻量级 `ONNX` 格式。
   - 包装为本地 HTTP REST 接口 (FastAPI)。
2. **基于 Prompt Choice List 的请求设计**:
   - 接口接收的 JSON Payload 应高度契合 `aiplayspire`：
     ```json
     {
       "current_state": { "deck": [...], "relics": [...], "gold": 120, "hp": 55, "floor": 12, "boss": "Slime Boss", "potions": [...] },
       "choices": [
         { "action": "pick_card", "target": "Strike_R" },
         { "action": "pick_card", "target": "Iron Wave" },
         { "action": "skip", "target": null }
       ]
     }
     ```
   - 后端逻辑：遍历 `choices`，推演结合 `current_state` 后的新状态 $S'$，将其输入 Value 模型。
   - 返回值：给出分数最高的具体选项。

## 潜在隐患与风险 (Pitfalls & Risks)
2. **药水库存状态追踪 (Potion Inventory)**：
   - 必须通过解析 JSON 中的 `potions_obtained` 和 `potions_floor_usage`，在重放器（reconstructor）中精确还原每个 Floor 开始时的“药水栏持有状态”（比如 `[Fire Potion, Empty, Empty]`），作为强特征输入。这能让 AI 准确评估那些为应对特定节点而购买的药水价值。由于微观战斗暴搜解决，这里的重点仅在于“让宏观 AI 在非战斗节点（如商店）具备花钱买临时战斗力（药水）的大局资产储备意识”。
3. **玩家行为噪音 (已通过A15+过滤缓解)**：
   - 高进阶（A20）和低进阶（A1）的拿牌逻辑天差地别（比如 A20 前期极度需求即战力，A1可以无脑抓体系牌）。现在依靠底层只选取 A15+ 玩家数据的策略，直接斩断了低端局的新手抓废牌噪音，模型不会被教出“不点 Skip (跳过)” 的毒瘤习惯。但仍需注意即便在 A15+，仍要合理调配胜场(Win)和负场(Loss)数据的利用率。
4. **延迟收益与当前血量的矛盾（金币/血量估值困难）**：
   - 神经网络有时很难量化“失去 10 点血，获得 100 金币”的真正价值，因为它需要理解“金币要在将来的商店才能变现”。如果距离商店还有 5 个层数，模型可能只看到了短期掉血的劣势而拒绝事件。这需要通过优化损失函数或长期折现率（类似于强化学习的 $\gamma$）来缓解。
5. **新隐患：极端短视主义 (Extreme Short-Sightedness)**：
   - 虽然将目标从“整局通关”改为“击杀当前 Act Boss”极大缓解了前期被贪死的问题，但也可能导致模型在第一章过度抓取前期牌（如完美打击），而排斥所有发挥慢的成长性卡牌（如恶魔形态）。到了 Act 2 初期可能会因卡组潜力不足而崩盘。**对策与备选方案**：
     - (1) 适当引入后期奖励：可在接近 Boss 的几个楼层（如 13-15层）稍微引入下一章初期的存活 Label 作为辅助奖励。
     - (2) **多任务输出备选 (Multi-Head Value Prediction)**：如果短视极其严重，可将最后一层改为输出包含长期视野的存活向量：$V(State) = [P(过Act1), P(过Act2), P(过Act3), P(碎心)]$。实机推演时通过配置不同系数的加权和来对选择打分，当前章权重最高，长期留有底单底线。目前 MVP 版本将先严格遵守“活过本章 Boss”的基线原则。
6. **新隐患：Set Transformer 实机推理性能瓶颈**：
   - 实机决策时（特别是商店或事件的蒙特卡洛多次推演），系统瞬间可能需要请求上百次 $V(S')$ 评估。Transformer 的 Attention 复杂度为 $O(N^2)$（此处 $N$ 为卡组+遗物总数，约 40-60 个 Token）。相较于原先极轻量的 MLP 架构，推理延迟可能从 1ms 涨到 20ms。**对策**：需严格控制 Attention 层数（2-4层级），并强制依赖 ONNX/TensorRT 进行 C++ 级底层加速，必要时在推演端建立 local caching（状态缓存）机制避开冗余运算。
